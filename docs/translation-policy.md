# 中文全文翻译与验收标准

## 完成定义

合格译文让数据库内核、查询处理和存储系统读者能够沿原论文结构完整阅读，不依赖整页截图、临时审校笔记或另行打开 PDF 才能补齐正文内容。准确性和完整性优先于文风润色。

## 进入条件

- 仅使用论文目录中的可读 `source.pdf` 作为正文证据。
- PDF 页数不得超过 `config/policy.yaml` 的全文翻译上限，除非其中已有用户明确授权的单篇例外。
- 已有 `reading_status: translated` 时，不无故重译。
- 缺失原文时使用 `unavailable`；页数超限或当前策略不翻译时使用 `skipped`，不猜正文。

## 正文要求

- H1 统一为 `# <paper.yaml 中的正式原文标题>（中文译文）`，避免猜造未获证据支持的中文书名。
- H1 后统一使用 `## 译者说明`，正文固定为：`本文依据同目录的 source.pdf 翻译。章节、图表、公式、算法、代码与参考文献按原文结构保留。`（文件名在 Markdown 中使用代码格式）。译者说明不得承载抽取、审校或 QA 过程记录。

- 保留原文首页可读的作者机构与联系信息、分类信息、摘要、章节、子节、脚注、结论、致谢、附录、收稿/修订记录和逻辑顺序；出版模板中的纯版权样板可不进入阅读正文，但不得静默丢弃承载技术定义、来源链接或时间信息的内容。
- 保留原文叙述人称：原文以 `I/we` 自述时分别译为“我/我们”，相应的 `my/our/us` 也保持第一人称，不得为了显得正式而改写成“作者”或“作者团队”。原文明示 `the author(s)` 时可译为“本文作者”；相关工作中的第三方作者应写明“该文作者”“文献 [N] 的作者”等指代对象，避免使用无法判断所指的裸“作者”。
- 术语采用数据库领域常用译法；重要术语首次出现可保留英文，后文保持一致。
- 公式保留语义、编号和引用关系；不能可靠转写时报告证据缺口。
- 参考文献不仅要逐条保留，正文中的引用锚点也要随对应论断完整保留。单篇深门禁会把源文正文出现而译文正文缺失的参考文献编号列为人工复核候选；区间、数组等方括号数字可能产生候选，必须回到 PDF 判断，不能机械豁免或机械补号。
- 正式表格保留编号和数值，优先转为 Markdown；复杂表格可附局部精裁图，但不得用整页截图替代。
- 图、架构图和流程图应精确裁剪，保留坐标轴、图例、标签、箭头和子图边界，并放在对应段落附近。裁图四边保留少量安全留白；译文已单独转写图注时不把原文图注或相邻正文收入位图。图片须在正常正文阅读宽度下保持最小标签可读，低清资源应从 PDF 渲染页重裁而不是放大旧图。正文内同一资源只引用一次。
- 原文有编号的 Listing/代码清单必须保留原编号、标题和完整正文，并与对应 fenced code block 相邻。仅保留“清单 N 展示……”、交叉引用或解释性摘要不算完成；原文可见的 `...` 或 `[...]` 省略标记也应原样保留，不自行补全作者省略的代码。
- 其他未编号的 SQL、代码、伪代码和配置也必须完整保留，使用 fenced code block 并保持可读格式；不能用执行过程摘要替代可见正文，也不能把原文代码改写成自拟的简化实现。
- 参考文献按原编号完整恢复为结构化列表，不使用“详见原文”或摘要列表代替。
- 最终阅读路径不得包含 TODO、contact sheet、候选裁图、覆盖索引、返工日志、自检表或“人工确认后通过”等过程残留。
- 完整性检查只认可读者可见的 Markdown。HTML 注释不得用于凑齐章节、中文覆盖、图表、算法、Listing 或参考文献；未闭合的 HTML 注释从起点至文件末尾都视为隐藏内容。代码围栏、缩进代码块和行内代码中的同形字符仍是读者可见的代码，不按注释处理。
- 正式代码围栏必须包含可读的实质性载荷，正式表格必须同时具有表头和数据行且列结构一致。透明、空白、近乎单色、尺寸过小或只改变极少像素的图片不算正式资源。
- 参考文献节必须由标题后的实际编号条目证明，不能把目录或页眉中的同名文字当成正文起点。页码范围、作者名和 URL 中的 PDF 换行断词必须恢复；悬空的数值范围（例如只有起点而无终点）属于确定性错误，不能 waiver。

## 可移植公式语法

译文使用仓库定义的 **GitHub-first 可移植子集**：GitHub 正确渲染是硬要求；同时采用主流 Markdown 阅读器普遍支持的分隔符和标准 TeX，以获得自然兼容性，但不为其他平台牺牲数学语义。AI 代理和译者只需遵守以下规则；平台怪癖及诊断细节由工具处理：

- **数学语义最高优先。** 不改动变量、大小写、上下标归属、数值、运算关系、分组和公式编号；无法确认等价性时停止并报告。
- **只用通用分隔符。** 行内公式写成 `$...$`；块公式的两个 `$$` 各自独占一行。不要使用数学代码围栏、`\(...\)`、`\[...\]` 或 GitHub 专有的反引号分隔形式。
- **编号公式不用 `\tag`。** 必须保留原公式编号及正文引用关系，并把编号作为自包含文本附在同一块公式末尾，例如 `\qquad \text{(2.1)}`。不得为了修复显示而删除编号或改变公式主体。
- **让公式成为独立的 Markdown 元素。** 不要把公式放进代码跨度、斜体、链接文字、脚注定义或图片 alt。公式与正文、标点或 Markdown 标记相邻时，按工具诊断添加 ASCII 空格或改写句子；不要自行发明兼容技巧。
- **表格公式不用 ASCII 管道符。** 公式内不写 `|` 或 `\|`；根据原义人工选择绝对值、范数或关系符号。无法判断含义时停止并报告。
- **字面标点使用 GitHub 可传递的受限 `\verb`。** 公式中确需显示字面 `_`、`%` 或 `#` 时，只能分别写成完整形式 `\verb0_0`、`\verb0%0` 或 `\verb0#0`。数字 `0` 分隔符会让下划线保持在 ASCII 单词内部，避免 GFM 把同一行多个字面下划线配成强调；这三种形式在普通公式和 Markdown 表格中都会原样进入 GitHub 公式节点并显示对应字符。不要使用会渲染失败的 `\char`，也不要使用会被 Markdown 消费的 `\_`、`\%`、`\#` 及其双重转义变体。
- **只写自包含的 TeX，让工具判定兼容性。** 不定义宏、不更改渲染器设置，也不自行使用字符编码、拆词或空节点等兼容技巧。提交前运行 `make math-check`；对变更译文运行 `make math-audit-github FILES='path/to/translation.md'`。只有工具明确给出安全候选时，才可显式运行 `make fix-math FILES='path/to/translation.md'`。

静态兼容错误在 `draft` 和 `translated` 状态下都直接失败。活跃 `draft` 还会拒绝包含数学运算符、完整渐近复杂度或下标向量的强确定性代码跨度候选；单变量、字段名等歧义内容仍必须由审阅者回到 PDF 判断，不能把扫描器未报告当作合法证明。规则编号、GitHub 原因、安全修复边界和外部审计步骤见 `docs/portable-math-maintainers.md`；这些检查不替代公式语义审校。VS Code、Obsidian、Typora、GitLab 等阅读器属于 best-effort 兼容对象，不得以它们的限制为由改坏原公式。

## 审校要求

以下要求规定单篇论文一旦进入翻译或审校后的质量标准；适用于新增、变更及已获授权复审的论文，不构成定期或全库历史复审授权，具体范围由 review workflow 和用户当前请求确定。

验收采用独立的“源文清单轮 + 逐项对照轮”。源文清单轮先盘点首页信息、所有章节、脚注、公式、表格、图、算法、代码、结论、致谢、附录、收稿记录和参考文献，再核对译文，不从译文目录反推原文是否完整；还要在恢复 PDF 换行断词、排除参考文献/代码/第三方引语后，逐项列出作者性 `I/we/my/our/us` 的上下文。逐项对照轮检查关键机制以及所有显式数字、单位、比较方向、故障边界、限定条件和结论；不能只抽查少量代表性数字。作者性自述须逐项核对叙述者和所有关系，即使中文自然共用一个主语，也不能把责任归属改成无主语、“本文”“系统”或“团队”；原始 token 数与“我/我们”数量只用于发现风险，数量相等不构成通过证据。对于双栏 PDF，应确认文本提取顺序；对于图表密集页、脚注密集页和抽取顺序可疑页，应渲染原页并视觉检查裁图边界、子图 `(a)/(b)` 标签、坐标刻度、轴标题、图例与正文交叉引用，不能以图片文件存在或数量相等代替图内语义核验。

字符数和中英文比例用于发现摘要化风险，不能替代逐节核对。摘要化候选的正文词数由仓库锁定版本的 pypdf 确定性抽取；原始计数和比例用于诊断，规则及严重度才构成验收时的语义发现。所有 Markdown 侧覆盖与结构统计先排除 HTML 注释。Poppler 布局抽取继续辅助 Listing、资源与参考文献扫描，但不同平台的布局差异不得被误当作新的已审阅对象。机械风险信号保留在校验输出和验收证据中，不进入单篇阅读元数据。

未完成逐节验收时使用 `reading_status: draft`；该状态允许校验器把 TODO、低覆盖、表格、Listing 和资源缺口报告为 warning。验收完成后，独立审阅者先生成绑定源文、译文、非忽略资源、翻译策略、受审书目信息、验收门禁、固定基线、完整检查项、发现记录及逐项 waiver 证据的 review receipt，再由 accept 校验同一份 receipt 后改为 `translated`；receipt 之后的任何受绑定输入变化都会阻止 accept。receipt 中的身份保证级别为 `self-attested`，用于记录实际分工而不是冒充密码学签名。未获精确处置的候选在 `translated` 中是硬失败。机械候选只扩大人工复审范围，不构成语义完整性证明。
